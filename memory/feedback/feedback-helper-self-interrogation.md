새로운 helper, 공통 함수, 추상화(hook/HOF/wrapper)를 만들기로 결정한 순간 — 코드 작성 직전 — 다음 체크리스트를 명시적으로 통과시켜 사용자에게 보고한다. 통과시키지 않고 헬퍼를 도입하면 안 된다.

**자기심문 체크리스트** (적용 가능한 항목만 골라서 답):
1. 정말 3+ 위치에서 반복되는가? (단순 2회는 헬퍼 회피, 인라인 유지)
2. 헬퍼 도입으로 줄어드는 라인 수 vs 새 추상화의 인지 비용 — 손익이 명확한가?
3. 헬퍼가 무엇을 **숨기는가**? 숨겨도 되는가? (silent 옵션, throw swallow, 기본값)
4. 시그니처가 너무 미래지향적이지 않은가? (지금 필요 없는 opts/generic)
5. 헬퍼 없이는 어떤 코드가 되나? 그게 정말로 더 나쁜가?
6. 헬퍼 도입으로 생기는 함정은? (잘못된 default가 호출 측에서 안 보임)
7. 이미 언어/라이브러리/프로젝트 내에 비슷한 게 있는가? 중복 추상화는 아닌가?
8. 헬퍼의 단위 테스트가 자연스럽게 작성되는가? (어렵다면 설계 결함 신호)

**Why**: 자동적으로 helper로 빼는 습관은 "premature abstraction" 안티패턴을 낳고, 잘못 설계된 추상은 호출부 모두를 망친다. 시스템 프롬프트의 "Don't add abstractions beyond what the task requires"와 일치하지만, brainstorming/설계 단계에서는 helper 제안이 매력적으로 보여 사용자가 직접 제동을 걸어야 했던 경험을 반복하지 않기 위함.

**How to apply**:
- design doc / spec / 브레인스토밍 섹션에서 helper 제안 직후 자기심문 결과를 인라인으로 제시
- 코드 작성 시점에 helper를 만들기로 결정하면 그 자리에서 위 체크리스트를 한 번 통과
- 통과한 결과(어떤 항목이 왜 OK인지)를 한두 줄로 사용자에게 보고 후 진행
- 통과 못 하는 항목이 있으면 helper 보류하고 인라인 유지 또는 설계 변경
- "이건 명백히 helper감"이라는 직관에 굴복하지 말 것 — 직관일수록 더 엄격하게 통과시킬 것
